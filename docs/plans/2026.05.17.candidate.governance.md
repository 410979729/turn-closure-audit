# Turn Closure Audit Candidate Governance Plan

> **For Hermes:** Use this plan before implementing candidate governance. Keep `turn-closure-audit` as the audit and governance companion; do not merge it into `scope-recall` and do not create a third plugin.

**Goal:** Upgrade `turn-closure-audit` from a simple post-turn audit logger into a clean append-only candidate-ledger and promotion workflow that makes every possible memory item reach an explicit terminal decision.

**Architecture:** `turn-closure-audit` observes each turn, redacts previews, classifies whether a candidate exists, records candidate events in an append-only ledger, and prepares dry-run or opt-in promotion through existing Hermes tools/providers. It must remain conservative: raw evidence and candidate drafts are not automatically equivalent to durable memory.

**Tech Stack:** Python 3.10+, append-only JSONL event log, existing Hermes plugin hooks, pytest, optional integration with `scope-recall` receipts.

---

## Product boundaries

### In scope

- Shared contract with `scope-recall`: candidate schema, final sinks, receipt schema, compatibility matrix.
- Structured candidate records with stable IDs and status.
- Append-only candidate event ledger as source of truth.
- Compatibility output for legacy daily review JSONL.
- Terminal states and receipts/rejection reasons.
- Dry-run promotion reports.
- Idempotent promotion decisions.
- Low-risk auto-promotion as explicit opt-in config, not default.
- Risk/confidence/final-sink metadata.
- Clear CLI/script/command surface for pending/report/promote.

### Out of scope

- No new plugin.
- No repository merge with `scope-recall`.
- No blind LLM writes to durable memory.
- No direct editing of user memories without gates.
- No storage of full raw conversations in candidate files.
- No secrets in candidates or reports.
- No treating review files, daily notes, or candidate ledgers as final memory sinks.

## Final sinks

A candidate's final sink is the intended terminal destination. Intermediate evidence locations are not final sinks.

Allowed final sinks:

```text
user
memory
project
ops
skill
knowledge
discard
ask_user
```

Not final sinks:

```text
memory/day
knowledge/review candidate
turn.closure.audit/*.json
turn.closure.audit/candidates/*.jsonl
general
```

Writing a candidate to a review JSONL or daily note does **not** count as promotion.

---

## Candidate lifecycle

Every candidate must eventually have a terminal state.

```text
pending
  -> promoted
  -> merged
  -> rejected_noise
  -> rejected_temporary
  -> rejected_sensitive
  -> needs_user_confirmation
  -> expired
```

No candidate should stay `pending` indefinitely. Reports must surface pending age and overdue counts.

## Phase 1 — Candidate schema

### Task 1.1: Add schema module

**Objective:** Make candidate records and events explicit and stable.

**Files:**
- Create: `candidate_schema.py`
- Test: `tests/test_candidate_schema.py`

**Schema sketch:**

```python
from __future__ import annotations

from dataclasses import dataclass, field
from typing import Any, Literal

CandidateStatus = Literal[
    "pending",
    "promoted",
    "merged",
    "rejected_noise",
    "rejected_temporary",
    "rejected_sensitive",
    "needs_user_confirmation",
    "expired",
]

FinalSink = Literal[
    "user",
    "memory",
    "project",
    "ops",
    "skill",
    "knowledge",
    "discard",
    "ask_user",
]

RiskLevel = Literal["low", "medium", "high"]

EventType = Literal[
    "candidate.created",
    "candidate.classified",
    "candidate.promote.dry_run",
    "candidate.promoted",
    "candidate.merged",
    "candidate.rejected",
    "candidate.needs_user_confirmation",
    "candidate.expired",
]

@dataclass
class CandidateRecord:
    candidate_id: str
    record_id: str
    session_id: str
    created_at: str
    updated_at: str
    status: CandidateStatus
    classification: str
    final_sink: FinalSink
    risk: RiskLevel
    confidence: float
    candidate_content: str
    evidence_summary: str
    decision_reason: str
    source: dict[str, Any] = field(default_factory=dict)
    receipt: dict[str, Any] | None = None
```

**Rules:**

- `candidate_content` must be redacted, concise, and suitable for promotion if approved.
- `evidence_summary` should be a redacted span/summary, not full transcript.
- `candidate_id` must be deterministic for dedupe: hash normalized `candidate_content + final_sink + classification`, with `record_id` kept as provenance only.
- Reject or downgrade candidates with empty `candidate_content`.
- Keep backward-compatible aliases only at serialization boundaries if older files used `recommended_sink`, `content`, or `evidence`.

**Tests:**

- schema roundtrip to/from dict;
- invalid final sink rejected;
- invalid status rejected;
- deterministic ID ignores superficial whitespace/case where appropriate;
- candidate_id changes when final sink changes.

## Phase 3 — Classification upgrade

### Task 3.1: Map classifications to final sink/risk/confidence

**Objective:** Give every candidate a final destination and risk.

**Files:**
- Modify: `classification.py`
- Test: `tests/test_turn_closure_audit.py`
- Test: `tests/test_candidate_schema.py` if helpers live there

**Initial mapping:**

```text
user-preference-or-boundary -> final_sink=user, risk=medium by default
explicit "remember"/"以后" preference -> final_sink=user, risk=low when no secrets/config
knowledge-governance mechanics -> final_sink=project or knowledge, risk=medium
task-outcome-or-diagnostic -> final_sink=knowledge or skill, risk=medium
interrupted-turn -> final_sink=ask_user, risk=high
routine -> no candidate or final_sink=discard with rejected_noise only when needed for audit
```

**Important:** Risk is not quality. Risk means danger of wrong/secret/over-broad permanent memory.

## Phase 4 — Promotion workflow

### Task 4.1: Add dry-run promotion engine

**Objective:** Decide what would happen without writing memory.

**Files:**
- Create: `promotion.py`
- Test: `tests/test_promotion.py`

**Inputs:**

- pending candidates;
- existing durable memory search results if available;
- config flags;
- final sink/risk/confidence;
- sensitivity/conflict checks.

**Outputs:**

```json
{
  "would_promote": [],
  "needs_confirmation": [],
  "suggest_skill": [],
  "suggest_knowledge": [],
  "would_reject": [],
  "conflicts": [],
  "already_satisfied": []
}
```

**Default decisions:**

- Explicit stable user preference, low risk -> recommend promotion; actual auto-promotion only if opt-in enabled.
- Workflow/procedure -> suggest skill/knowledge, not memory row.
- Temporary service state -> reject_temporary.
- Secrets or credential-shaped text -> reject_sensitive.
- Conflicting facts -> needs_user_confirmation.
- Existing equivalent memory -> merged or already_satisfied.

---

### Task 4.2: Make promotion idempotent

**Objective:** Running promotion twice must not duplicate memories or candidate terminal events.

**Files:**
- Modify: `promotion.py`
- Modify: `candidate_ledger.py`
- Test: `tests/test_promotion.py`

**Rules:**

- Promotion by `candidate_id` checks current materialized candidate status first.
- If already terminal, return existing receipt/status.
- Before writing to `scope-recall`, search or dedupe against existing durable rows.
- Same candidate content + final sink should result in one durable memory or one merge receipt, not duplicates.

**Acceptance:** Two runs of the same promote command produce one durable action and stable follow-up no-op/receipt behavior.

## Quality bar

- Candidate decisions are explicit and auditable.
- Every candidate can reach a terminal state.
- Append-only event log is the source of truth.
- Legacy review files remain readable but are not truth.
- No hidden writes.
- No secrets in candidate records.
- No giant runtime.py.
- Tests cover schema, state transitions, idempotency, receipts, and compatibility output.
- Public docs are honest and conservative.
